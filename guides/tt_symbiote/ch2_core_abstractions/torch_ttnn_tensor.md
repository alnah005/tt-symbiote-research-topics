# `TorchTTNNTensor` — a `torch.Tensor` subclass with a TTNN backend

`TorchTTNNTensor` (defined in `models/experimental/tt_symbiote/core/tensor.py`) is the data carrier of TT Symbiote. By subclassing `torch.Tensor` it remains a first-class citizen in every framework that inspects tensor types, while its internal plumbing routes compute to TTNN hardware.

## Dual backing store

A `TorchTTNNTensor` holds at most one live backing at a time:

| Attribute | Type | Carries live data when |
|---|---|---|
| `elem` | `torch.Tensor` | the tensor has not yet been uploaded, or after it has been brought back to CPU |
| `ttnn_tensor` | `ttnn.Tensor` | the tensor is resident on a Tenstorrent device |

The invariant is soft: both can technically be non-`None` during a transition, but normal operations ensure only one is the canonical source of truth at any moment. `clone` and (via `to_torch`) `tolist` and `numpy` honour this by checking `ttnn_tensor` first. `shape` does not — it checks `elem` before falling back to the raw `ttnn_tensor` path (see the `shape` reporting section below for the full priority order).

`elem` is declared in `__slots__`, so it is always present as a slot attribute. `ttnn_tensor` is set by the run-implementation layer (`TENSOR_RUN_IMPLEMENTATION`) and is not declared directly in `tensor.py`.

## Construction

```python
t = TorchTTNNTensor(elem_or_ttnn_tensor, dtype=torch.float32)
```

`__new__` delegates immediately to `TENSOR_RUN_IMPLEMENTATION.new_instance`, which means the actual allocation strategy is controlled by the active run configuration (default TTNN, CPU fallback, etc.), not by this file. This is the same indirection used by `TTNNModule.__call__`.

## `__torch_dispatch__` — the interception hook

Every time PyTorch executes an ATen-level operation (add, matmul, view, slice, …) on a `TorchTTNNTensor`, Python calls `__torch_dispatch__` before any CPU kernel runs:

```python
@classmethod
def __torch_dispatch__(cls, func, types, args=(), kwargs=None):
    return TENSOR_RUN_IMPLEMENTATION.torch_dispatch(cls, func, types, args, kwargs)
```

`TENSOR_RUN_IMPLEMENTATION.torch_dispatch` is where the dispatcher is consulted. The typical implementation calls `can_dispatch_to_ttnn(func_name, args, kwargs)` and, if `True`, hands off to `dispatch_to_ttnn`. If `False`, it falls back to executing `func` on the `elem` backing.

Because `__torch_dispatch__` is a classmethod on `torch.Tensor`, it catches operations regardless of whether the tensor is an argument, a keyword argument, or the result of a previous dispatch. This makes it the correct hook for transparent backend substitution — it fires for every ATen call, including ones generated by higher-level PyTorch APIs.

## Explicit conversion properties

Two properties expose on-demand conversion between the two backends:

| Property | Direction | Behaviour |
|---|---|---|
| `to_ttnn` | CPU → device | delegates to `TENSOR_RUN_IMPLEMENTATION.to_ttnn(self)` |
| `to_torch` | device → CPU | delegates to `TENSOR_RUN_IMPLEMENTATION.to_torch(self)` |

These are properties, not methods — call them without parentheses:

```python
ttnn_t = my_tensor.to_ttnn   # uploads to device if needed
cpu_t  = my_tensor.to_torch  # downloads from device if needed
```

`tolist()` and `numpy()` both call `self.to_torch` internally, so they always produce CPU data regardless of where the tensor currently lives.

## `shape` reporting

The `shape` property has three cases, evaluated in order:

1. **Distributed tensor with a live TTNN backing** — if `ttnn_distributed_tensor_config` is set and `ttnn_tensor` is not `None`, `shape` calls `ttnn_distributed_tensor_config.get_logical_shape(self.ttnn_tensor.shape)`. This translates the per-shard physical shape stored on each device back to the logical shape the user expects.

2. **No active distributed TTNN backing — CPU-side elem present** — if `elem` is not `None`, returns `self.elem.shape`. This branch fires when no `DistributedTensorConfig` is attached, but also when one is attached and `ttnn_tensor` is `None` (CPU-backing state). In the latter case `get_logical_shape` is NOT called, so a distributed tensor in CPU-backing state returns raw `elem.shape`, not the logical shape.

3. **No active distributed TTNN backing — TTNN tensor present** — falls back to `tuple(int(i) for i in self.ttnn_tensor.shape)`, converting TTNN's shape representation to a plain Python tuple. This case implicitly requires `ttnn_tensor` to be non-`None`. Like case 2, this branch can be reached by a distributed tensor if `ttnn_distributed_tensor_config` is `None`; it does not call `get_logical_shape`.

> **Crash condition:** If both `elem` and `ttnn_tensor` are `None` when case 3 is reached, the expression `self.ttnn_tensor.shape` raises `AttributeError`. There is no graceful fallback. The dual-backing invariant — at least one of `elem` or `ttnn_tensor` must carry live data at any time — is a hard requirement, not a soft guideline. A tensor where both are `None` is an invalid state and will crash on any `shape` access.

In summary: only case 1 (distributed tensor with a live TTNN backing) returns the logical shape via `get_logical_shape`. Cases 2 and 3 return raw `elem.shape` or raw TTNN shape respectively, regardless of whether a `DistributedTensorConfig` is attached. Callers should not assume that attaching a `DistributedTensorConfig` alone causes `shape` to report the logical shape — the TTNN tensor must also be live.

## `DistributedTensorConfig` attachment

For tensors that are sharded across a device mesh, a `DistributedTensorConfig` can be attached:

```python
tensor.set_distributed_tensor_config(distributed_tensor_config)
```

The config is stored in `self._distributed_tensor_config` (in `__dict__`, not in `__slots__`) and exposed through the read-only property `ttnn_distributed_tensor_config`. When `None`, the tensor is treated as single-device.

`DistributedTensorConfig` (defined in `models/experimental/tt_symbiote/core/run_config.py`) is a dataclass with three fields:

| Field | Purpose |
|---|---|
| `mesh_mapper` | Maps the logical tensor onto the device mesh during upload (`ttnn.to_device`) |
| `mesh_composer` | Reassembles shards back into a single tensor during download (`ttnn.from_device`) |
| `logical_shape_fn` | Optional callable that converts the per-shard TTNN shape to the full logical shape reported by `shape` |

If `logical_shape_fn` is `None`, `get_logical_shape` returns the TTNN shape unchanged.

> **Note:** `set_distributed_tensor_config` is typically called by `TTNNModule.set_output_tensors_config`, not directly by user code. The module calls it on its output tensors based on `self.device_state.tensor_config`, so the correct mapper/composer pair propagates automatically.

## Arithmetic operator overloads

`TorchTTNNTensor` overloads the standard Python arithmetic dunder methods (`__mul__`, `__rmul__`, `__sub__`, `__rsub__`, `__add__`, `__radd__`, `__abs__`, `__matmul__`, `__rmatmul__`) by routing them through the corresponding `torch.*` functions. This ensures that expressions like `a + b` or `a @ b` go through `__torch_dispatch__` rather than being resolved by Python's default numeric protocol, which would bypass the dispatch hook.

## `clone`

```python
new_t = tensor.clone()
```

If `ttnn_tensor` is set, clones it with `ttnn_tensor.clone()`. Otherwise clones `elem`. The result is a new `TorchTTNNTensor` with the same `dtype`.
