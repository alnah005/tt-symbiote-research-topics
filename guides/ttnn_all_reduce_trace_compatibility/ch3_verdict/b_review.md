# B Review — Chapter 3: Trace Compatibility Verdict and Requirements — Pass 1

1. **`use_composite_reduce_scatter` logic is inverted in Q1 (factual error — wrong implementation outcome).**
   `q1_trace_compatibility.md` states: "`use_composite_reduce_scatter` returns `true` (forcing composite) when: The scatter dimension size does not divide evenly by `num_devices`". The actual code (`composite_common.cpp` lines 46–48) returns `false` when the scatter dimension is not evenly divisible — i.e., non-divisibility suppresses the composite path, not enables it. The real condition that returns `true` is when the scatter dimension IS divisible by `num_devices` **AND** the per-device slice is not tile-aligned. As written, a reader would conclude that an uneven scatter dimension forces the composite (trace-incompatible) path and would raise a false alarm or misplace a guard assertion. The actual consequence is that an uneven scatter dimension falls back to the native (non-composite) path. The bullet in the "conditions" block must be removed or rewritten.

2. **Q2 misattributes `ttnn.experimental.all_reduce_async` to `TTNNLinearIColShardedWRowSharded` (factual error — wrong op named).**
   `q2_semaphore_state.md` introduces the contrast section with "used, for example, in `TTNNLinearIColShardedWRowSharded.forward`" as the representative caller of `ttnn.experimental.all_reduce_async`. The actual `TTNNLinearIColShardedWRowSharded.forward` (`linear.py` lines 158–172) calls `ttnn.experimental.reduce_scatter_minimal_async` directly — there is no call to `ttnn.experimental.all_reduce_async` anywhere in `linear.py`. A reader following this attribution would look for an `all_reduce_async` call in that class, not find it, and either doubt the analysis or incorrectly assume the two ops are synonymous.

3. **Composite path intermediate count is wrong for this use case in Q1 (factual error — wrong number).**
   `q1_trace_compatibility.md` states the composite path allocates "three dynamic intermediates — the interleaved conversion tensor, the reshaped tensor, and the `gather_tensor`". However, the interleaved conversion tensor (`sharded_to_interleaved`) is only allocated when `change_mem_config` is `true`, i.e., when the input is sharded (`all_reduce_async.cpp` lines 279–284). `TTNNLinearIColShardedWAllReduced` always passes `memory_config=ttnn.DRAM_MEMORY_CONFIG` to `ttnn.linear`, so `input_is_sharded` is `false` and the conversion tensor is never created even on the composite path. The composite path for this specific caller allocates two transient buffers (`reshaped_tensor` and `gather_tensor`), not three. A downstream integrator counting buffers to pre-allocate would over-provision.

4. **Q3 Requirement 5, item 1 conflates two independent guards (misleads implementer).**
   `q3_requirements_and_limitations.md` states that passing `memory_config=ttnn.DRAM_MEMORY_CONFIG` "prevents the `composite_for_2d_mesh` path via sharded input". `DRAM_MEMORY_CONFIG` prevents `sharded_to_interleaved` being called (a separate code path), but it has no effect on the `composite_for_2d_mesh` guard, which checks only `GetFabricConfig() == FABRIC_2D && is_true_2d_mesh(input_tensor)` — fabric and mesh shape, not memory config. A reader following the requirement list might believe that setting an interleaved memory config is a sufficient substitute for confirming `FABRIC_1D`, and skip the separate fabric check (Requirement 3).

5. **Navigation footer missing on `index.md` (structural gap).**
   `index.md` has no navigation footer pointing back to Chapter 2 or forward to Chapter 4. The other three content files all end with a `**Next:**` footer. The reading order section in `index.md` lists the files but provides no backward link to Chapter 2's index and no forward link to Chapter 4, leaving the chapter orphaned in both directions for readers who land on the index page directly.

# B Review — Chapter 3 — Pass 2

1. **Composite path mechanics are materially wrong throughout (Q1 and `index.md`) — wrong implementation outcome.**
   The guide consistently describes the composite path as "all-gather + `moreh_sum`" (index.md table row for Q1, Q1 conclusion section, Q1 path-selection section, Q3 Limitation 1). The actual cluster-axis overload (`all_reduce_async.cpp` lines 295–323) calls `composite_common::composite_all_gather` followed by `local_sum` / `local_sum_float32`. `composite_all_gather` internally uses `all_broadcast` + `concat` — not a primitive `ttnn::all_gather`. `moreh_sum` is only used inside `local_sum` for non-float32 dtypes; float32 inputs use `local_sum_float32` which calls `ttnn::sum` via `transpose`. A reader trying to profile which kernel ran, or to replicate/extend the composite path, would look for an `all_gather` kernel and a `moreh_sum` kernel and find neither in the expected form. The characterisation must be corrected to "all_broadcast + concat (composite_all_gather) + local reduce (moreh_sum for non-float32, ttnn::sum for float32)".

2. **`use_composite_reduce_scatter` conditions block is internally contradictory (Q1) — reader would mismodel the logic.**
   The "returns `true`" bullet list (lines 66–70 of q1) correctly states the two conditions that force composite. The next paragraph (lines 72–75) then states: "It returns `false` … when the scatter dimension size does **not** divide evenly by `num_devices`." The source (`composite_common.cpp` lines 46–48) shows this is an *early return* that exits before the real force conditions are evaluated — so the structure of the function is: early-exit-false (non-divisible) → check row-major → check tile alignment. The guide presents non-divisibility as a third category alongside the two true-return conditions, rather than as a guard that short-circuits all of them. An implementer building a path-prediction routine from this description would construct the wrong decision tree (e.g., treating divisible + tile-aligned as ambiguous instead of guaranteed non-composite).

3. **Requirement 2a code example demonstrates the wrong overload for the stated goal (Q3) — implementer would call wrong function.**
   Requirement 2a in q3 presents a Python snippet that calls `ttnn.experimental.all_reduce_async(tt_output, scattered_buffer, ...)` and labels `scattered_buffer` the "stable intermediate" for the `reduce_scatter` step. This maps to the overload at `all_reduce_async.cpp` line 426 that takes `buffer_tensor` as a second positional argument and routes directly to `ttnn::prim::all_reduce_async` — a single-step fused op that does not use the reduce-scatter + all-gather two-step path at all. The `buffer_tensor` in that overload is the intermediate for the fused op, not the `scattered_tensor` of the non-composite two-step path. An integrator following this example as a drop-in replacement for `ttnn.all_reduce` would invoke a fundamentally different execution path (fused single primitive vs. two separate collectives) with different latency, bandwidth, and semaphore requirements, and would not actually have a pre-allocated `scattered_tensor` for the reduce-scatter step.
