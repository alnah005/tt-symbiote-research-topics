# B Review — Chapter 4 — Pass 3

1. **`integration_checklist.md` lines 146–149: "routes the result through `entry.trace_output`" is unsupported by the code and contradicts the correct description two sentences above.**
   Lines 140–144 correctly state: "The warm-up output buffer and the capture-window output buffer are distinct allocations." Lines 146–149 then claim "replays update the device-side memory associated with the capture-window output buffer, and the framework routes the result through `entry.trace_output` for the caller." No such routing exists in `run_config.py`. `ttnn.execute_trace` writes into the capture-window buffer; `entry.trace_output` (the warm-up buffer, at a different device address) is never updated. A reader implementing against this description would expect `entry.trace_output` to hold the latest replay result and would read stale warm-up data on every subsequent call. The claim should be removed or replaced with an accurate statement that the capture-window buffer, not `entry.trace_output`, is what `execute_trace` writes into.

2. **`minimal_test_pattern.md` Step 5 assigns `trace_output` from inside the capture window — inconsistent with `_capture_trace` and with `integration_checklist.md` lines 140–144, leading to a wrong mental model of the framework.**
   The skeleton at Step 5 writes `trace_output = layer.forward(trace_input)` after `ttnn.begin_trace_capture`. The actual `_capture_trace` (line 990 of `run_config.py`) runs the warm-up call *before* `begin_trace_capture`, discards the capture-window output with `_ = ...`, and stores the warm-up output as `TraceEntry.trace_output`. The skeleton's `trace_output` therefore refers to the capture-window buffer (which `execute_trace` does update), while the framework's `TraceEntry.trace_output` refers to the warm-up buffer (which `execute_trace` does not update). The mapping table row "Step 5 — `trace_output` stored in `TraceEntry`" equates these two different buffers. A reader building a `TracedRun`-equivalent from this skeleton will expect that keeping `entry.trace_output` alive is sufficient to observe replay outputs — it is not in the real framework, because replays write to a different buffer.

# B Review — Chapter 4 — Pass 4

1. **`minimal_test_pattern.md` Step 4b uses `persistent_input` before it is defined — causes `NameError` for any implementer following the steps in order.**
   Step 4b (lines 164–165) calls `warm_up_output = layer.forward(persistent_input)`, but `persistent_input` is not allocated until Step 5 (lines 189–193). The skeleton is presented as an ordered sequence that the reader "translates into a production test". Following the numbered steps in order produces a `NameError` on `persistent_input` at Step 4b. This will cause an incorrect implementation for any reader who copies the pattern sequentially.

2. **`minimal_test_pattern.md` Step 6 aliasing check always passes trivially and cannot detect actual aliasing — misleads the reader into false confidence.**
   Both `output_1` and `output_2` in Step 6 are read from `warm_up_output`. As confirmed in `run_config.py` lines 992–1000, `ttnn.execute_trace` writes into the capture-window output buffer (the buffer allocated inside `begin_trace_capture` / `end_trace_capture`); `warm_up_output` (the warm-up buffer stored as `TraceEntry.trace_output`) is never updated by any replay call. Both reads will therefore return the same stale warm-up data regardless of whether semaphore aliasing or buffer overlap is present. The check `assert torch.allclose(output_1, output_2)` will always pass, providing zero diagnostic signal. A reader implementing this as their aliasing detector will miss real aliasing bugs. The comment in Step 6 acknowledges the limitation but still labels the block as an "aliasing check", which is materially misleading.

3. **`integration_checklist.md` aliasing detection code block has the same non-functional check — always passes, cannot detect aliasing.**
   The post-capture aliasing detection code block (lines 196–213 of `integration_checklist.md`) reads `entry.trace_output` twice after two `ttnn.execute_trace` calls. For the same reason as item 2 above — `execute_trace` never writes into the warm-up buffer (`entry.trace_output`) — both reads return the same stale warm-up tensor. The `assert torch.allclose(output_1, output_2)` will always succeed. A reader implementing this post-capture validation step as their aliasing check will get a permanently green test that fails to catch any real aliasing.

4. **`minimal_test_pattern.md` Step 7 numerical comparison is non-functional — `output_1` contains warm-up data, not replay data, so PCC always passes trivially.**
   `output_1` in Step 7 is taken from the `ttnn.to_torch(warm_up_output, ...)` call in Step 6. As established above, `warm_up_output` holds the result of the non-traced warm-up forward pass, not the result of any replay. `reference_output` (Step 3) is also produced by a non-traced forward pass over the same input. Both tensors come from the same computational path with the same input, so `comp_pcc(reference_output, output_1, pcc=0.999)` will return a PCC of approximately 1.0 regardless of whether the trace produces numerically correct results. The check provides no evidence that the traced path is numerically correct and will give implementers false confidence that trace validation has passed.
