# B Review — Chapter 1: GlobalSemaphore Internals — Pass 5

1. [global_semaphore_api.md, Warning block under `ttnn.reset_global_semaphore_value`] The Warning states: "it does not wait for a prior in-flight op to finish — if a CCL kernel is still using the semaphore when reset is called, both the previous op and the reset write will contend for the same L1 location." This is factually wrong for the standard usage path. `reset_global_semaphore_value` calls `EnqueueWriteMeshBuffer(..., blocking=true)` on the mesh command queue. If the caller follows the natural Python call order — CCL op is dispatched first (enqueued to the CQ), then `reset_global_semaphore_value` is called — the reset write is enqueued into the same CQ after the CCL dispatch. Because the CQ is a FIFO, the CCL kernels on device will complete before the reset write executes on device; there is no contention. The described race ("both will contend for the same L1 location") can only occur if `reset_global_semaphore_value` is called on a separate command queue that bypasses the CCL dispatch CQ, or if the reset is called before the CCL dispatch is enqueued at all. The current Warning would mislead an implementor into believing that extra explicit synchronization (e.g., a `Finish()` or host-side wait) is required before every `reset_global_semaphore_value` call — unnecessary overhead and a conceptually wrong model of the CQ. Fix: Revise the Warning to accurately state that in single-CQ usage, FIFO ordering guarantees the CCL kernels complete on device before the enqueued reset write executes; the caller's actual responsibility is to not call `reset_global_semaphore_value` before the CCL dispatch has been enqueued (i.e., before the Python CCL op call returns), and to not use a separate out-of-order CQ for the reset.

The rewritten block (now a Note) accurately states that in the standard single-CQ pattern used by `TT_CCL`, the CQ's FIFO ordering guarantees the reset write executes only after all prior-enqueued CCL kernels have completed on device — no extra `Finish()` or barrier call is required. The real constraints (do not call reset before the CCL dispatch is enqueued; do not use a separate out-of-order CQ) are clearly stated, and multi-CQ or direct L1 write scenarios are scoped out as non-standard usage.
