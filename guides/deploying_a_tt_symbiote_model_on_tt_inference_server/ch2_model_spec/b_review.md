## Pass 1

**1. `max-num-batched-tokens` formula contradicts the T3K example (factual error — wrong numerical answer)**

`adding_a_model_spec.md` states the value is "typically `max_context * max_concurrency`". The T3K example uses `max_concurrency=32` and `max_context=131072`, so the formula yields `131072 * 32 = 4,194,304`. However, the example actually sets `"max-num-batched-tokens": "131072"` — the same value as `max_context` alone, not the product. A reader applying the formula would produce a value ~32× larger than the example, and the formula would also imply the N300 example (concurrency=1) is special-cased, not general. Either the formula or the T3K example value is wrong; both cannot be correct simultaneously.

**2. `MODEL_SPECS` lookup key is `model_id` but `workflow_commands.md` says `--model` matches `model_name` (factual error — incorrect implementation)**

`schema_reference.md` defines `ModelSpec.model_id` as "the unique key" and says specs are keyed by `model_id` in the `MODEL_SPECS` dictionary. `workflow_commands.md` says the `--model` argument "is matched against the `model_name` field" — a different field defined as "a short slug derived from the HF repo name". The example command uses `--model my-symbiote-model-8b`, which looks like a `model_name` slug, not a `model_id` (which would be `id_tt_symbiote_my-symbiote-model-8b_n300`). A reader building a lookup or testing from the CLI cannot tell which value to pass, and will likely try the wrong key.

**3. `ModelSpecTemplate` expansion gap: `spec_templates` list is never connected to `MODEL_SPECS` (critical structural gap — incorrect implementation)**

`adding_a_model_spec.md` Step 3 instructs the reader to append to a `spec_templates` list. `schema_reference.md` says the `MODEL_SPECS` dictionary "is generated by expanding a set of `ModelSpecTemplate` instances at import time" via `expand_to_specs()`. The docs never state whether the reader must also call `expand_to_specs()` on the list, whether this happens automatically, or where and how the two are wired together. A reader following the steps as written cannot determine whether their template will actually appear in `MODEL_SPECS`.

**4. `perf_reference` field referenced in workflow but absent from `DeviceModelSpec` schema (factual error — incorrect implementation)**

`workflow_commands.md` states that the `reports` workflow performs "pass/fail validation against the `perf_reference` targets defined in the `DeviceModelSpec`". The `DeviceModelSpec` dataclass in `schema_reference.md` has no `perf_reference` field. A reader adding a `DeviceModelSpec` will omit this field, and then not understand why the reports workflow produces no pass/fail output (or why validation silently passes everything).

**5. `ModelSpecTemplate` does not include `status`, `docker_image`, `uses_tensor_model_cache`, or `has_builtin_warmup`, but `ModelSpec` requires them — no explanation of how they are set (critical structural gap — incorrect implementation)**

`schema_reference.md` describes `ModelSpec` as having `status`, `docker_image`, `uses_tensor_model_cache`, and `has_builtin_warmup` fields. `schema_reference.md` also advises readers to set `status=ModelStatusTypes.Experimental` for new integrations. However, these fields do not appear in `ModelSpecTemplate`, which is the only thing the reader constructs. With no explanation of where these values come from during expansion, a reader cannot produce a valid `ModelSpec` and cannot follow the `Experimental` → `Functional` guidance.

## Pass 2

**1. `ModelSpecTemplate` dataclass definition omits `uses_tensor_model_cache` and `has_builtin_warmup` but instructions say to set them on the template (factual error — incorrect implementation)**

Both `schema_reference.md` (the `ModelSpecTemplate` note) and `adding_a_model_spec.md` (the `uses_tensor_model_cache` / `has_builtin_warmup` section) tell readers to "set the corresponding flags on `ModelSpecTemplate` before expansion." However, the `ModelSpecTemplate` dataclass definition shown in `schema_reference.md` does not include these fields — only `weights`, `impl`, `tt_metal_commit`, `vllm_commit`, `inference_engine`, and `device_model_specs` appear. A reader who tries to set `uses_tensor_model_cache=True` on `ModelSpecTemplate` will get a `TypeError` (unexpected keyword argument), because the field is not declared on the dataclass. Either the dataclass definition in the schema reference must be updated to include these fields, or the instructions must describe an alternative mechanism (e.g., a post-expansion mutation).

**2. `workflow_commands.md` lists only four `--device` values, omitting all Blackhole device types (factual error — incorrect implementation)**

`workflow_commands.md` states that `--device` accepts `n150`, `n300`, `t3k`, `galaxy` (case-insensitive). The `DeviceTypes` enum in `schema_reference.md` also defines `P100`, `P150`, `P150X4`, `P150X8`, and `P300`. A reader deploying a symbiote model to Blackhole hardware has no documented CLI value to pass for `--device` and will receive an unresolved model error or guess incorrectly (e.g., passing `p150b` or `blackhole`). The accepted device values in `workflow_commands.md` must include all `DeviceTypes` enum members.

**3. N300 `max-num-batched-tokens` example silently uses `max_context` instead of the decode formula, with no explanation (factual error — wrong numerical answer for multi-sequence N300 configs)**

The `adding_a_model_spec.md` N300 example sets `"max-num-batched-tokens": "32768"` (equal to `max_context=32768`) with `max_concurrency=1` and `block_size=64`. The decode formula from both `schema_reference.md` and `adding_a_model_spec.md` is `max_concurrency * block_size`, which for N300 would be `1 * 64 = 64`, not `32768`. The example produces a value 512× larger than the formula. No explanation is given for why the prefill-heavy path was chosen for this example. A reader who increases `max_concurrency` on N300 (e.g., to 4) while following the example will set `max-num-batched-tokens` to `32768` instead of the decode-correct `4 * 64 = 256`, over-allocating the forward-pass budget and potentially causing out-of-memory errors on hardware.

## Pass 4

**1. `uses_tensor_model_cache` / `has_builtin_warmup` ownership is stated contradictorily across two files (factual coherence error — incorrect implementation)**

The `ModelSpecTemplate` dataclass definition in `schema_reference.md` (lines 136–141) explicitly includes `uses_tensor_model_cache` and `has_builtin_warmup` as named fields with defaults. Yet the Note immediately following that definition (`schema_reference.md`, line 146) states these are "fields on the *expanded* `ModelSpec` instance, not on `ModelSpecTemplate`." `adding_a_model_spec.md` (line 135) repeats the same claim: "fields on the *expanded* `ModelSpec` instance (not on `ModelSpecTemplate` itself)." A reader seeing the dataclass code knows the fields ARE on `ModelSpecTemplate`, but then reads prose that says they are NOT — with no reconciliation. A reader who trusts the prose over the code will look for an alternative way to set these flags, fail, and conclude the docs are broken. A reader who trusts the code will set the fields correctly but may still distrust their understanding of the architecture. The prose in both files must be corrected to match the dataclass definition: these fields are declared on `ModelSpecTemplate` and propagated to each expanded `ModelSpec` during `expand_to_specs()`.

## Pass 6

**1. `ModelStatusTypes` threshold definition contradicts the `perf_reference` mechanism (factual error — materially wrong conceptual understanding)**

`schema_reference.md` defines the `ModelStatusTypes` enum with explicit percentage-of-peak thresholds: `Functional` = "Achieves at least 10% of theoretical peak performance" and `Complete` = "Achieves at least 50% of theoretical peak performance." However, the `perf_reference` section in the same file describes an entirely different mechanism: the `reports` workflow compares measured values against the absolute numeric targets in `perf_reference` (e.g., `1200.0` tok/s) to determine `ModelStatusTypes`. `workflow_commands.md` confirms this, stating that `reports` generates "pass/fail validation against the `perf_reference` targets defined in the `DeviceModelSpec`." The two descriptions are irreconcilable: a reader trying to set a `perf_reference` that promotes a model to `Functional` cannot determine whether the target should be 10% of some hardware-spec-derived peak throughput number, or simply any absolute value the reader chooses. A reader who trusts the enum definitions will attempt to calculate theoretical peak before writing `perf_reference`, then find that the reports workflow ignores that calculation entirely.

## Pass 5

**1. `ModelSpec` serialized JSON contains no `impl_id` field, but the server startup description claims it uses `impl_id` for model registration (factual coherence error — incorrect implementation)**

`schema_reference.md` documents `ModelSpec` as having these fields: `model_id`, `hf_model_repo`, `model_name`, `device_type`, `device_model_spec`, `env_vars`, `uses_tensor_model_cache`, `has_builtin_warmup`, `docker_image`, and `status`. There is no `impl` or `impl_id` field on `ModelSpec`. However, the "How `TT_MODEL_SPEC_JSON_PATH` is used" section (same file) states that `run_vllm_api_server.py` "Registers the model's implementation class with vLLM's `ModelRegistry` using `impl_id` to locate the correct module." Since `impl_id` does not exist as a field in the serialized `ModelSpec` JSON, a reader debugging a registration failure — or writing a tool that parses the JSON — has no documented way to determine which field carries `impl_id` or how the server resolves it. The documentation must either add `impl_id` (or the full `ImplSpec`) as a field on `ModelSpec`, or explain how `impl_id` is extracted from an existing field (e.g., parsed from the `model_id` string pattern `id_{impl_id}_{...}`).

A reader who sets `perf_reference` entries and expects the `reports` workflow to automatically graduate the model from `Functional` to `Complete` cannot determine what values to write or how to encode tier membership. The graduated status path described in the enum definitions is unimplementable from the documentation as written.
