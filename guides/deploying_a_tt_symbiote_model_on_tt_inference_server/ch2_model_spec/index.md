# Chapter 2 â€” The ModelSpec Configuration System

Every model that runs through `tt-inference-server`'s workflow automation must have a corresponding `ModelSpec` entry. The `ModelSpec` is the single source of truth for a model-device pair: it carries the HuggingFace repository reference, the pointer to the implementation code, all device-specific vLLM launch flags, hardware tuning overrides, environment variable injections, and the Docker image tag that was validated against those settings. Without a `ModelSpec` registration, `run.py` has no way to resolve a model name to a concrete serving configuration, and the Docker container has no way to know which vLLM arguments to apply. Understanding the schema is therefore the first concrete integration step for any new model â€” including a TT Symbiote model whose implementation lives in a separate repository.

## Why this chapter matters

`tt-inference-server`'s workflow orchestration layer resolves every model reference by looking up a name in the `MODEL_SPECS` dictionary, which is generated by expanding a set of `ModelSpecTemplate` instances at import time. Each template combines one or more HuggingFace weight repositories with one or more `DeviceModelSpec` entries through a Cartesian product, producing one fully instantiated `ModelSpec` per (weight, device) pair. The vLLM launch script then reads the serialized `ModelSpec` JSON at the path given by `TT_MODEL_SPEC_JSON_PATH` and uses it to construct the vLLM command line. Getting the schema right â€” field names, types, and valid values â€” is therefore a prerequisite for everything that follows: server launch, benchmarking, evaluation, and report generation all depend on the same spec entry.
