# B Review — Chapter 1: Quantization Formats on Wormhole — Pass 1

1. **bfloat4_b_format.md, line 25** — Incorrect precision count. The text states "3 bits mantissa" with an "implicit leading 1" yields "4 levels of precision between consecutive powers of 2." With 3 explicit mantissa bits plus the implicit leading 1, the significand has 4 bits, encoding 2³ = 8 distinct values between consecutive powers of 2, not 4. A reader implementing a quantization check or estimating rounding error will get the wrong answer. Fix: replace "4 levels" with "8 levels (2³)" in the precision description.

2. **bfloat8_b_format.md, line 23** — Per-element layout diagram omits the sign bit, contradicting the index summary. The ASCII diagram shows `[MMMMMMMM]` (8 mantissa bits, no sign), but index.md line 53 states bfloat8_b has "~7" mantissa bits (implying a sign bit consumes the remaining bit). If each element has 1 sign bit + 7 mantissa bits = 8 bits total, the diagram is wrong. If truly 8 mantissa bits with no per-element sign, then index.md's "~7" mantissa claim and the sign column entry "(shared)" are wrong. The inconsistency will cause implementors to miscount bits. Fix: align the diagram and index table to the actual Tenstorrent hardware spec — either `[S | MMMMMMM]` (1 sign + 7 mantissa) or `[MMMMMMMM]` (8 mantissa, sign implicit/shared) with a clear note in both places.

3. **index.md, line 51 / bfloat8_b_format.md conceptually** — The format comparison table lists the bfloat8_b sign bits field as "(shared)", implying the sign bit is shared across the tile like the exponent. Sign bits are not shared in standard block floating-point — only the exponent is shared; each element retains its own sign. If this is intentional Tenstorrent behavior, it needs an explicit statement; if it is a copy-paste error from the exponent row, it is factually wrong and will mislead a reader computing per-element storage. Fix: correct the sign column for bfloat8_b and bfloat4_b to show "1 (per element)" unless hardware documentation explicitly confirms shared sign semantics.

4. **bfloat4_b_format.md, line 189 / precision step example** — The step-size example at line 188–189 states `0.5 / 2^3 = 0.0625` as the step size for bfloat4_b. With 3 explicit mantissa bits, the resolution between consecutive representable values at scale 0.5 is `0.5 / 2^3 = 0.0625`, which is numerically correct as written. However, the conclusion "0.001 rounds to 0.0 (below the minimum representable relative value)" is only true if the step size at magnitude 0.001 exceeds 0.001, which requires the shared exponent to be set by the 0.5 value — a conditional the text does not make explicit. More critically, the minimum representable non-zero value in the tile is not zero; it is `0.5 × 2^(−(2^3−1))` for the denormal case, or simply the smallest normal step. The example as written can mislead a reader into thinking any value smaller than 0.0625 rounds to zero, which is incorrect — only values smaller than half the step size do. Fix: add a sentence clarifying that 0.001 rounds to zero *because* 0.001 < 0.0625/2 = 0.03125 (half the step size), rather than implying any value below 0.0625 vanishes.

5. **bfloat16_format.md, line 36** — Decimal precision for bfloat16 is stated as "~2.4 digits." The number of significant decimal digits for a format with `m` mantissa bits (including the implicit leading 1) is `(m+1) × log10(2)`. For bfloat16, m = 7 explicit bits → 8-bit significand → 8 × log10(2) ≈ 8 × 0.301 ≈ 2.41 digits. This is correct. However, bfloat4_b_format.md line 48 also claims "~2.4 digits (relative)" for bfloat8_b with "~7 (tile-relative)" mantissa bits — but bfloat8_b has at most 7 explicit mantissa bits (same as bfloat16), so the table in bfloat4_b_format.md incorrectly equates bfloat16 and bfloat8_b decimal precision. bfloat16 has a self-contained 8-bit significand; bfloat8_b has 7–8 mantissa bits *shared-exponent relative*, making their effective per-element decimal precision roughly the same *only within a tile where magnitudes are similar*. The table does note "(relative)" but does not warn that across tiles bfloat8_b may be coarser. This is a conceptual gap that could cause a reader to incorrectly conclude bfloat8_b and bfloat16 always have equivalent precision. Fix: add a footnote to the bfloat4_b comparison table clarifying that bfloat8_b's ~2.4 digits applies only intra-tile; inter-tile dynamic range is limited by the per-tile shared exponent, unlike bfloat16's per-element exponent.
