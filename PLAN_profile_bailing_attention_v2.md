# Plan v2: Fix Tracy DRAM Buffer Overflow for BailingAttention Profiling

**Date:** 2026-03-26
**Problem:** Device profiler DRAM buffer overflow at ~12,000 markers when profiling 32 decode iterations of BailingAttention. Markers are dropped, resulting in incomplete/corrupt device profiling data.

---

## Root Cause Analysis

### How the profiler DRAM buffer works

The profiler allocates per-RISC DRAM banks to store markers. The buffer size is controlled by:

```
DRAM buffer per RISC = (L1_MARKER_SIZE * (PROGRAM_ID_COUNT + GUARANTEED_MARKER_COUNT + optional_markers)) * program_support_count
```

**Key constants** (from `hostdevcommon/profiler_common.h`):
- `PROFILER_L1_MARKER_UINT32_SIZE = 2` (each marker is 2 uint32s = 8 bytes)
- `PROFILER_L1_PROGRAM_ID_COUNT = 2`
- `PROFILER_L1_GUARANTEED_MARKER_COUNT = 4`
- `PROFILER_L1_OPTIONAL_MARKER_COUNT = 250`

**Default program support count:** 1000 (from `profiler_state_manager.cpp` line 19)

When the `DROPPED_ZONES` bit is set in the control buffer, the profiler.cpp code (line 1504-1516) emits a warning:
> "Profiler DRAM buffers were full, markers were dropped! ... Please either decrease the number of ops being profiled or run read device profiler more often"

### Why 32 decode iterations overflow

Each decode iteration generates ~17-20 device ops (matmuls, all_gather, to_memory_config, RoPE, paged_sdpa, etc). Each op generates markers on multiple RISCs (BRISC, NCRISC, TRISC0/1/2). With 32 decode iterations + 1 prefill + warmup, that is roughly:

- 33 profiled forward passes x ~20 ops/pass x ~5 RISCs x ~2 markers/RISC = ~6,600 marker writes per core
- Plus warmup iterations add more

The buffer fills up because markers accumulate across ALL program dispatches until the profiler data is read back by the host (which happens at process exit or explicit dump).

## Recommended Approach: Option C (Both -- Reduce Iterations + Increase Buffer)

### Change 1: Reduce `num_decode_tokens` from 32 to 8

**File:** `models/experimental/tt_symbiote/tests/test_profile_bailing_attention.py`

**Change:** Line 129: `num_decode_tokens = 32` -> `num_decode_tokens = 8`

**Rationale:**
- 8 decode iterations still provides steady-state data (warmup runs 3 iterations before the profiled section)
- Reduces marker count by ~4x, well within buffer limits
- Still enough iterations for statistical averaging
- The original 32 was chosen for timing stability, but device profiling only needs 5-8 iterations for reliable per-op cycle data

### Change 2: Set `TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT` to increase DRAM buffer

**Environment variable:** `TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT=2000`

**Source:** `tt_metal/llrt/rtoptions.cpp` line 827-833

```
// TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT
// Increase the profiler DRAM buffer to support more programs.
// Usage: export TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT=500
case EnvVarID::TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT: {
    this->profiler_program_support_count = std::stoi(value);
}
```

Default is 1000. Setting to 2000 doubles the DRAM profiler buffer. The buffer consumes DRAM space calculated as:

```
per_risc = marker_size * (6 + optional) * program_count
total_per_dram_channel = per_risc * 5_riscs * ceil(96_cores / 12_channels) = per_risc * 5 * 8 = per_risc * 40
```

At 2000 program support count, this is still a modest fraction of the 1GB DRAM per channel on Wormhole.

**NOTE:** Increasing `TT_METAL_PROFILER_PROGRAM_SUPPORT_COUNT` also increases `PROFILER_FULL_HOST_BUFFER_SIZE_PER_RISC`, which is passed as a compile-time define. This means the build must be compiled with profiling enabled (which it already is when using `TT_METAL_DEVICE_PROFILER=1`).

## How to Get the FULL ops_perf_results CSV with Op Names

### Understanding the output files

The profiler generates TWO CSV files:

1. **`profile_log_device.csv`** -- Raw per-core, per-RISC marker data (timestamps, zone IDs, core coords). This is low-level and does NOT contain op names by default.

2. **`cpp_device_perf_report.csv`** -- Higher-level per-program report with columns:
   - `GLOBAL CALL COUNT` -- runtime_id (sequential op counter)
   - `METAL TRACE ID` / `METAL TRACE REPLAY SESSION ID`
   - `DEVICE ID`, `DEVICE ARCH`
   - **`OP NAME`** -- The TTNN op name (e.g., `ttnn::matmul`, `ttnn::all_gather`)
   - `CORE COUNT`, `AVAILABLE WORKER CORE COUNT`
   - `DEVICE TRACE FIRMWARE DURATION [ns]`
   - `DEVICE TRACE KERNEL DURATION [ns]`
   - `DEVICE KERNEL FIRST TO LAST START [ns]`
   - `DEVICE FW DURATION [ns]`, `DEVICE FW START CYCLE`, `DEVICE FW END CYCLE`
   - `DEVICE KERNEL DURATION [ns]`, `DEVICE KERNEL START CYCLE`, `DEVICE KERNEL END CYCLE`
   - Per-RISC kernel durations: `DEVICE BRISC/NCRISC/TRISC0/TRISC1/TRISC2/ERISC KERNEL DURATION [ns]`
   - `OP TO OP LATENCY [ns]`, `OP TO OP LATENCY BR/NRISC START [ns]`

This file is generated by the C++ post-processing code in `profiler_analysis.cpp` (function `writeProgramsPerfResultsToCSV`), using the analysis configs from `cpp_device_analyses.json`.

### Getting the cpp_device_perf_report.csv

This report is generated automatically when:
1. `TT_METAL_DEVICE_PROFILER=1` is set (enables Tracy profiler build)
2. `TT_METAL_PROFILER_CPP_POST_PROCESS=1` is set (enables C++ post-processing)

The output directory is:
- `$TT_METAL_PROFILER_DIR/.logs/` if `TT_METAL_PROFILER_DIR` is set
- Otherwise: `$TT_METAL_HOME/generated/profiler/.logs/`

### What about FPU utilization and NOC bandwidth columns?

The current C++ analysis (`cpp_device_analyses.json`) only computes duration-based analyses (`PROGRAM_FIRST_TO_LAST_MARKER` type). It does NOT compute:
- FPU utilization
- NOC bandwidth
- PM ideal (performance model ideal time)

These metrics are NOT available in the built-in C++ post-processing. To get utilization metrics, you would need:
- The Tracy GUI (which shows device zones with cycle counts that you can manually cross-reference with known op FLOP counts)
- Custom analysis scripts that read `profile_log_device.csv` and compute utilization from cycle counts and known hardware specs

For the purpose of this profiling task, the `cpp_device_perf_report.csv` provides the essential data: **per-op device kernel durations with op names and op-to-op latencies**.

## Expected CSV Output Structure

### cpp_device_perf_report.csv Columns

```
GLOBAL CALL COUNT,METAL TRACE ID,METAL TRACE REPLAY SESSION ID,DEVICE ID,DEVICE ARCH,OP NAME,CORE COUNT,AVAILABLE WORKER CORE COUNT,DEVICE TRACE FIRMWARE DURATION [ns],DEVICE TRACE KERNEL DURATION [ns],DEVICE KERNEL FIRST TO LAST START [ns],DEVICE FW DURATION [ns],DEVICE FW START CYCLE,DEVICE FW END CYCLE,DEVICE KERNEL DURATION [ns],DEVICE KERNEL START CYCLE,DEVICE KERNEL END CYCLE,DEVICE KERNEL DURATION DM START [ns],DEVICE KERNEL DM START CYCLE,DEVICE KERNEL DM END CYCLE,DEVICE BRISC KERNEL DURATION [ns],DEVICE NCRISC KERNEL DURATION [ns],DEVICE TRISC0 KERNEL DURATION [ns],DEVICE TRISC1 KERNEL DURATION [ns],DEVICE TRISC2 KERNEL DURATION [ns],DEVICE ERISC KERNEL DURATION [ns],OP TO OP LATENCY [ns],OP TO OP LATENCY BR/NRISC START [ns]
```

### Key Columns for Analysis

1. **OP NAME** -- Identifies the TTNN op (e.g., `ttnn::matmul`, `ttnn::all_gather`, `ttnn::paged_sdpa_decode`)
2. **DEVICE KERNEL DURATION [ns]** -- Total kernel execution time across all cores (max of all core durations)
3. **DEVICE BRISC/NCRISC/TRISC KERNEL DURATION [ns]** -- Per-RISC durations to identify compute-bound vs memory-bound ops
4. **OP TO OP LATENCY [ns]** -- Host dispatch gap between consecutive ops (this is the "dispatch overhead")
5. **CORE COUNT** -- Number of cores used by the op

### What to Look For

- **High OP TO OP LATENCY:** Indicates host dispatch bottleneck; candidate for trace capture
- **TRISC durations >> BRISC/NCRISC:** Compute-bound op
- **NCRISC duration >> TRISC:** Memory/NOC-bound op
- **Small DEVICE KERNEL DURATION with high OP TO OP LATENCY:** Op is dispatch-dominated; needs trace capture or fusion
